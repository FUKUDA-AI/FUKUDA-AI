Version: v1.0（設計のみ・実装なし）
最終更新日: 2026-07-06
状態: Released（CEO承認 2026-07-06・Morning Brief第1号・判断1）
前提文書: [AGENT_DESIGN.md](AGENT_DESIGN.md)（各Agentの個別設計）/ [../00_MASTER/ARCHITECTURE.md](../00_MASTER/ARCHITECTURE.md)

> 10のAgentは単独のツールではなく、**一つのOS**として動く。
> 原則: 情報は共有し、判断は一本化し、実行はCEOが握る。

---

## 1. Agent間の情報の流れ（全体像）

```
【現場層】 so u AI / SUNNY NOMADO AI / 催事AI / 営業AI / 広告・SEO AI
              │  日々の対応Draft・現場の気付き
              ↓
【資源層】 発注・在庫AI ⇄ 資金繰りAI （モノとカネの制約を返す）
              │  実行可能性・資金インパクト
              ↓
【中枢層】 CEO補佐AI （集約・調停・一本化）← 秘書AI（記録・期限・キュー管理）
              │  意思決定メモ（5点セット）
              ↓
【承認】   CEO（全最終判断）
              │  承認・却下・指示
              ↓
【還流】   Decision Log / Insight Draft → 学習サイクル（Pattern→Lesson→Principle→EP）
```

流れの原則は3つ。**上り**（現場→資源→中枢→CEO: 提案は必ず資源層の制約チェックを経てCEOへ）、**下り**（CEO→中枢→各Agent: 決定はCEO補佐AIが各Agentへ展開し、秘書AIが記録）、**還流**（全Agentの知見→Insight Draft→学習サイクル→EP/Knowledgeとして全Agentへ戻る）。

## 2. 起点になるAgent（トリガー別）

| トリガー | 起点Agent | 例 |
|---|---|---|
| 顧客からの接触 | so u AI / SUNNY NOMADO AI / 営業AI | 問い合わせ・手紙・見積依頼 |
| 商機・出店機会 | 催事AI / 営業AI | 会場の出店打診・卸の引き合い |
| 内部アラート | 発注・在庫AI / 資金繰りAI | 在庫不足・資金逼迫の検知 |
| データの変化 | 広告・SEO AI | 広告効率の悪化・急上昇 |
| アイデア・仮説 | 商品企画AI / CEO | 新商品の種・CEOの相談 |
| 定例スケジュール | 秘書AI | 日次・週次・月次の定例フロー起動 |

起点Agentは「一次情報の整理」までを担い、判断が必要なものは必ずCEO補佐AI経由でCEOへ上げる。

## 3. CEOへ報告するAgent

- **CEO補佐AI** — 唯一の定常報告窓口。全Agentの提案を集約・調停し、意思決定メモとして一本化して報告する（バラバラの報告でCEOの時間を奪わない）
- **秘書AI** — 報告の「形式」担当。日次サマリー・レビュー待ちキュー・期限リマインドを提示
- **例外（直訴ルート）**: 資金繰りAIの資金アラート、催事AI/so u AIの事故・クレーム発生時は、CEO補佐AI経由を待たず即時にCEOへ通知してよい（緊急時例外）。事後にCEO補佐AI・秘書AIが記録を追完する

## 4. Agent同士が共有するデータ

| 共有データ | 場所 | 内容 |
|---|---|---|
| Released Knowledge | 01_Knowledge（knowledge_index.json経由） | 全Agent共通の判断根拠 |
| EVOLVING/CORE Principles | 00_MASTER | 全Agent共通の判断原則 |
| 実績データ | 07_Data（将来: Shopify/催事/在庫/広告/財務） | 常時最新を参照可 |
| Agentレポート | 06_Reports | 各Agentの提案・分析（相互参照可） |
| 作業状態 | 10_AI_Memory | 進行中Sprint・保留・レビュー待ち |
| SOP | 04_SOP | 標準手順（released分） |

## 5. Agent同士が直接書き換えてはいけないデータ

| データ | 理由 | 変更ルート |
|---|---|---|
| 00_MASTER全文書（憲法〜CHANGELOG） | 思想・原則はCEOのみ改定 | CEO直接編集 or CEO承認Sprint |
| Released Knowledge（各カテゴリ直下） | 品質はCEOレビューが担保 | _drafts → 昇格フローのみ |
| knowledge_index.json | 自動生成物 | Knowledge Builderの再生成のみ |
| 各種ログ（insight/decision/pattern/lesson/principle_log.json） | 学習サイクルの原本 | 各Generator + CEOレビュー反映のみ |
| 他Agentのレポート（06_Reports） | 発信元の責任を保つ | 参照のみ。異論はCEO補佐AIへ提起 |
| 07_Dataの実績データ | 事実の原本 | データ取込パイプラインのみ |
| 10_AI_Memory | 作業状態の一貫性 | 秘書AI（またはセッション中のFUKUDA AI本体）のみ更新 |

## 6. CEO確認が必須になる判断（全Agent共通リスト）

1. **お金が動く判断すべて** — 発注確定・支払・借入・投資・広告予算・出店契約
2. **価格に関わる判断** — 値付け・見積の確定提示・送料等の条件変更
3. **ブランド表現の新パターン** — 新しい語り口・コラボ・新販路
4. **顧客への約束** — 納期確約・特別対応の発動・クレーム対応方針
5. **商品化の決定** — 憲法第8条四問の最終判定
6. **人・取引先に関わる判断** — 新規取引先・職人/工場の選定・契約条件
7. **思想・原則・Knowledgeの変更** — EP/CORE昇格・Knowledge released化（昇格フロー）
8. **対外発信の実行** — メール送信・SNS投稿・広告出稿（AgentはDraftまで）

## 7. 定例連携フロー

### 日次（毎朝）— CEO Morning Brief

```
秘書AI: 今日の予定・期限・CEOレビュー待ちキューを整理
  → 各Agent: 3行以内の日次ミニレポート提出（「異常なし」も1行報告）
  → CEO補佐AI: 判断が必要な事項を最大3件に絞り、1枚のMorning Briefへ統合
  → CEO: 5分で判断（承認/却下/保留 + 一言）
  → 秘書AI: Decision Log Draft記録・EP運用記録追記
```

詳細設計は [../06_Reports/CEO_MORNING_BRIEF_DESIGN.md](../06_Reports/CEO_MORNING_BRIEF_DESIGN.md) を正とする。

### 週次（週初）

```
各Agent: 週次レポートを06_Reportsへ（実績・気付き・Insight Draft候補）
  → 発注・在庫AI + 資金繰りAI: 今週のモノ・カネ制約サマリー
  → CEO補佐AI: 週次経営サマリー（数値+判断事項+EP運用記録の候補）
  → CEO: 週次レビュー（30分想定）
  → 秘書AI: 決定事項をDecision Log Draftへ、Sprint Close 3点セット更新
```

### 月次（月初）

```
CEO補佐AI: 月次総括（ブランド別・チャネル別の実績と学び）
  → 学習サイクル実行: 新規会話・対応ログからInsight/Pattern/Lesson/Principle抽出
  → CEOレビュー: EP候補・Knowledge Draft・EP運用記録の棚卸し
  → EVOLVING→CORE昇格候補の検討（複数ブランド有効性の確認・CEOのみ決定）
  → ROADMAP・Architecture の見直し
```

## 8. 典型シナリオ（6件）

### シナリオ1: 催事出店判断

```
① 催事AI（起点）: 会場から出店打診を受領 → 会場条件を整理
② 催事AI: KN-EVT-0001（平均30万）+ EP-005で一次評価。付帯コストも算入
③ 発注・在庫AI: 出店に必要な在庫の可否を回答
④ 資金繰りAI: 出店費用と回収見込みの資金インパクトを回答
⑤ CEO補佐AI: ①〜④を意思決定メモ（5点セット）へ一本化
⑥ CEO: 出店可否を決定 ★CEO確認必須（契約・費用）
⑦ 秘書AI: 決定をDecision Log Draftへ記録。催事AIがKN-SOP-0003で設営準備
⑧ 事後: 実績をレポート化 → 催事Knowledgeの更新Draft + Insight Draftへ還流
```

### シナリオ2: so u問い合わせ対応

```
① so u AI（起点）: お客様から手紙つきの注文・問い合わせを受領
② so u AI: KN-SOP-0001に従い、返信Draft + 職人への共有メモDraftを作成（EP-002）
③ 通常対応 → CEO（または担当者）が確認して送信 ★対外送信はAgent実行不可
④ 特別対応が必要な場合（通常外の製作希望）: EP-003を根拠に特別体制案を作成
   → CEO補佐AI経由でCEOへ ★特別対応の発動・価格はCEO確認必須
⑤ 秘書AI: 納期をリマインド管理。対応記録は顧客対応Knowledgeの更新Draftへ
```

### シナリオ3: 在庫不足判断

```
① 発注・在庫AI（起点）: 在庫アラート（催事予定に対して商品Xが不足見込み）
② 発注・在庫AI: 追加発注案を作成（数量・納期・根拠。リードタイム制約を明示）
③ 催事AI: 出店計画側の調整案（商品構成の変更）を代替案として提示
④ 資金繰りAI: 発注資金の影響を回答
⑤ CEO補佐AI: 「追加発注 vs 構成変更 vs 機会損失許容」を比較表で一本化
⑥ CEO: 決定 ★発注確定はCEO確認必須
⑦ SUNNY NOMADO受注生産品の場合: 「注文をもとに生産数量を決定」の原則を優先し、
   見込み発注の提案はしない（AGENT_DESIGN §5禁止事項）
```

### シナリオ4: 新商品企画

```
① 商品企画AI（起点）: 顧客の声・催事の反応・職人の技術情報から企画の種を整理
② 商品企画AI: 憲法第8条・四問チェックを一次実施（1つでも「いいえ」なら中止提案）
③ ブランドAI（so u / SUNNY NOMADO）: ブランド適合と発信ストーリーの観点で意見
④ 発注・在庫AI: 初回数量とリードタイム / 資金繰りAI: 開発・生産資金
⑤ CEO補佐AI: 企画書Draft（四問判定+資源制約つき）へ一本化
⑥ CEO: 四問の最終判定と商品化決定 ★CEO確認必須
⑦ 開発中: EP-007（品質は反復で確保・価格を約束しない）を全Agentが遵守
⑧ 完成後: 広告・SEO AIがローンチ文案Draft、営業AIが卸提案Draft
```

### シナリオ5: 広告改善判断

```
① 広告・SEO AI（起点）: 広告効率の変化を検知（データ接続後）/ 当面はCEO相談起点
② 広告・SEO AI: 改善案を作成。ただし値引き訴求・煽りは選択肢に入れない（EP-004・憲法第5条）
③ ブランドAI: 表現がブランドの文法（憲法第6・7条）に合うか確認
④ 資金繰りAI: 予算変更の資金影響
⑤ CEO補佐AI: 改善案を一本化（クリエイティブ変更 / 配分変更 / 停止 の選択肢）
⑥ CEO: 予算・新表現の承認 ★CEO確認必須
⑦ 効果測定結果はマーケティングKnowledgeの更新Draftへ還流
```

### シナリオ6: 事故・クレーム発生（緊急時例外）

```
① 現場Agent（催事/so u等・起点）: 事故・重大クレームを検知
② 【直訴ルート】CEO補佐AI経由を待たず即時CEOへ通知（§3例外）
③ 現場Agent: 事実の時系列整理 + 一次対応案Draft（推測を混ぜない）
④ CEO: 対応方針を決定 ★クレーム対応方針はCEO確認必須
⑤ 対応完了後（EP-006）: 謝罪で終わらせず、プロセスの見直しを必ず実施
   → 該当SOPの改訂Draft + Lesson候補として学習サイクルへ還流
⑥ 秘書AI: 一連の記録をDecision Log Draft + PENDINGへ登録
```

## 9. v1.0運用上の注意

- データソース（Shopify/Meta/在庫/財務/カレンダー）未接続のAgentは、該当ステップを「CEOからの手動情報提供」で代替する。**不足データはAgentが明示する**（推測で補わない）
- 外部データの接続設計・優先順位・段階分け（v1.1/v1.2/v2.0）は [../07_Data/DATA_SOURCE_DESIGN.md](../07_Data/DATA_SOURCE_DESIGN.md) を正とする
- 本連携設計は全Agent実装前の青写真である。実装はAGENT_DESIGN §12の優先順位（①CEO補佐 ②催事 ③so u）で進め、実装済みAgentから本フローに組み込む
- Agent間の「連絡」の実体は、v1.0では06_Reports上の文書と索引参照である（メッセージング機構は将来設計）
