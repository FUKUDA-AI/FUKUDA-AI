## 0. 基本思想（CEO指示 2026-07-08・本書の憲法）

> **「AIが昨夜までに会社を整理し、CEOは朝に判断だけをする。」**

- Morning Briefは**AIの状態報告ではない**。**CEOが朝5分で会社を動かすためのBrief**である
- Morning BriefはAIのためではなく、**CEOのための画面**である
- 目的: CEOが今日、**何を判断し / 何を任せ / 何を放置し / 何を確認するか**を5分で決められること
- CEOが知りたい順番: **会社の状態 → 今日判断すること → AIができること → 結果確認 → AI開発状況**。AI開発情報は最後で十分
- **FOSは単なるタスクリストではない。CEOのOperating Systemである。** Brief生成前にAIが必ずFOSをレビューする（レビューは提案であり、AIは勝手に変更しない）

## 1. v1.0からの転換

| | v1.0（現行） | **v2.0** |
|---|---|---|
| 中心 | AIが選んだ判断3件 | **会社の状態→判断→任せる→確認の流れ全体** |
| 順番 | 判断→やらないこと→レビュー待ち | **会社状態が最初・AI情報が最後** |
| FOS | 取込んで候補抽出 | **取込む前にAIがレビューし、改善提案を返す** |
| AIの役割 | 判断材料の整理 | 整理 + **「AIが今すぐ終わらせられる仕事」の提示（承認制）** |
| 売上・数値 | 注意セクションに変化のみ | **Company Statusとして常設**（接続7ソース統合・未接続明示） |

## 2. v2.0のセクション構成（⓪+7・追加仕様 2026-07-08反映）

```
💬 AIから一言      秘書の5行報告（最上段・CEO承認時追加 2026-07-08）
⓪ 昨夜のAI作業    AIが夜の間に終わらせた整理（3〜6行の短い報告）
① Company Status   会社の状態（売上・利益・現金・在庫・出荷）
② Event Status     催事の状態（催事がある日だけ数値表示）
③ CEO Decision     今日の判断（S→A→B・Briefの中心）
④ FOS Review       AIによるFOSレビュー結果（提案のみ）
⑤ AI Actions       承認すればAIが実行できる仕事
⑥ Result Review    結果確認（Result Layer・現行維持）
⑦ AI System        AI自身の情報（最後）
```

### 💬 AIから一言（最上段・秘書の型）

**Briefの一番最初**に、5〜6行でCEOが状況を掴める「秘書の一言」を置く。この5行だけで社長は状況が分かり、その後に詳細を見る。

構成の型（この順・各1行）:

```
おはようございます。
昨夜は異常ありません。          ← 夜間の異常有無（必ず言う）
Airレジを同期しました。          ← 実施した同期・取込（事実のみ）
催事は昨日ありません。           ← 催事の有無（0円とは言わない）
Shopifyは未接続です。           ← 未接続・未確定の注意（あれば）
今日はso uの発注判断が最重要です。 ← 今日の最重要判断1件（③の先頭と一致）
```

- ルール: **実行した事実と③の先頭判断のみ**で構成（推測・演出・煽りなし。静けさと余白=憲法第6条はBriefにも適用）
- 最重要判断は③CEO Decisionの並び順先頭（S→期限切れ→スタッフ待ち…）と必ず一致させる（一言と本文の食い違い禁止）

### ⓪ 昨夜のAI作業（AIから一言の直後・追加仕様）

- 目的: **朝起きた瞬間に「AIが夜の間に会社の整理を終わらせてくれた」状態**をつくる
- 内容例: Dashboard更新 / Dataset同期状況（Airレジ・Shopify…）/ CSV取込状況 / Learning更新 / Result更新 / メール下書き_件 / **異常なし**（沈黙と正常を区別）
- ルール: **実際に実行した事実のみ**を列挙（やっていないことを書かない・推測禁止）。3〜6行の短い報告に限定し、詳細は⑦AI Systemへ。これは「AI開発情報」ではなく**夜間オペレーションの完了報告**であり、冒頭配置は思想（AI開発情報は最後）と矛盾しない

### ① Company Status（最初に会社全体）

| 項目 | データ源 |
|---|---|
| 昨日売上 / 今月売上 / 前年比 | Shopify・MakeShop・Airレジ |
| 利益 | Shopify・FLAM（原価接続後） |
| 現金・入金予定・支払予定 | Airペイ・会計（接続後） |
| 在庫異常 | FLAM・はぴロジ・logiec |
| 出荷遅延 | はぴロジ・logiec |

- **接続7ソース（Shopify / MakeShop / Airレジ / Airペイ / FLAM / はぴロジ / logiec）から取得できる内容を統合**する
- **未接続は「未接続」と表示。推測は禁止**（現時点の接続済み=Airレジのみ。他6ソースは「未接続」表示から始まる）
- 数値はDataset Registryでactiveのソースのみ・機械的集計のみ（意味づけしない・sales_definition等の未確定定義は明示）

### ② Event Status（催事専用・新設）

- **催事がない日は「昨日は催事はありませんでした」と表示する。売上0円とは表示しない**（データがないことと売上ゼロを混同しない）
- 催事がある場合のみ表示: **催事売上 / 平均日商 / TOP商品 / 催事名 / 未確定事項**
- データ源: Airレジ（SalesRecord・催事照合済み分）+ 07_Data/events/（EventRecord）
- 「催事がある日」の判定: EventRecord会期 or Airレジdaily_salesの該当日データ存在（機械的判定のみ）

### ③ CEO Decision（Briefの中心）

- 表示順: **decision_importance S → A → B**（C・完了済みは載せない）
- 同ランク内の優先: **期限切れ → スタッフ待ち → 契約 → お金 → 発注**（FOS Operating Rule §6の並び順をこの優先領域で更新）
- **AI開発タスクは最後**（会社の現場判断が常に先）
- 各判断の5点セット（推奨・理由・効果・リスク・手順）+ 根拠EP/KN、CEO記入欄、未分類/未設定の1タップ確認 — v1.0の細則を継承
- 最大3件ルールは維持（S超過時のみ例外=FOS Operating Rule §6-2どおり）

### ④ FOS Review（新設・Brief生成前の必須工程）

**Morning Brief生成前に、AIがFOS全体をレビューする。** 検出する6観点:

| 観点 | 例 |
|---|---|
| 順番がおかしい | 期限が近いのに下位にある |
| 今日やる必要がない | 期限・依存から見て後ろ倒し可能 |
| AIが代行できる | メール下書き・CSV取込・レポート作成 |
| 人待ち | waiting_personがいるのに動いていない |
| 期限切れ | deadline超過（完了忘れの疑い含む） |
| 似たタスクが重複 | 同じ目的のタスクが複数箇所にある |

- 結果は**「AIからの提案」**として表示する。例: 「このタスクは明日でも問題ありません」「この2件はまとめられます」「メールはAIが作成できます」「このタスクはWaitingへ移動できます」
- **AIはFOSを書き換えない。提案だけする**（FOS変更・完了化・移動・統合はすべてCEO操作）
- 提案には根拠を添える（期限・依存・重複箇所。推測の断定はしない）

### ⑤ AI Actions（承認すればAIが実行できる仕事）

- **AIだけで完了できる仕事を一覧化**し、CEOは承認だけする
- 例: メール下書き / CSV取込 / Dashboard更新 / Result更新 / 議事録整理 / 契約書ドラフト / データ同期 / 売上レポート作成
- **制約（AI_CHARTER準拠・変えない）**: AIが「実行」できるのは**Draft作成・取込・生成・整理まで**。対外送信・支払・発注・FOS変更・Knowledge released化は引き続きCEOのみ（「メール作成」=下書きまで。送信はCEO）
- 抽出元: ⑤-1 FOSの **ai_ready属性**（§3）+ ⑤-2 FOS Reviewの「AIが代行できる」検出分
- 表示形式: `[ ] タスク名（所要目安・出力先）→ CEO: [ 承認 / 不要 ]`

### ⑥ Result Review（現行維持）

- Result Layer（result_recorder → 結果確認待ち → CEO判定: 成功/失敗/継続観察）を**そのまま維持**
- Result Layer v1.1（Action/Business 2層化・設計済み）採用時はv1.1仕様に従う

### ⑦ AI System（最後）

- Dataset（Registry・同期状態）/ Learning（6段階件数・増分）/ Dashboard / Result Layer / Knowledge の状態
- AI開発のレビュー待ち・開発系判断もここ（会社判断と混ぜない）

## 3. FOS「AI Ready」属性（CEO提案 2026-07-08・採用設計）

FOSに **ai_ready** 属性を追加し、「AIが今すぐ終わらせられる仕事」をBriefへ自動抽出する。

| 項目 | 内容 |
|---|---|
| 属性名 | `ai_ready`: **yes / no**（未入力=null=未判定） |
| 入力者 | CEO・スタッフ（FOS入力時）。**AIはai_ready=yesの候補を提案できるが、確定しない**（FOS Review提案として提示） |
| yesの目安 | メール返信作成 / CSV取込 / データ同期 / Dashboard更新 / 契約書ドラフト / 売上レポート作成 など、**Draft・取込・生成・整理で完結する仕事** |
| noの目安 | 対外送信・支払・発注・交渉・現場作業など、人の実行・判断を含む仕事 |
| Briefへの流れ | fos_importer が ai_ready を TaskRecord へ透過（推測で埋めない）→ Brief §5「AI Actions」へ自動掲載 → **CEO承認 → AIが実行（Draft/取込/生成）→ 完了報告 → FOS完了化はCEO操作** |
| 意味 | FUKUDA AIを「タスク管理システム」から**「実際に仕事を進めるAI秘書」**へ進化させる一歩 |

反映先（承認後の実装Sprint）: FOS Operating Rule **v1.3**（テンプレート15項目化=+ai_ready）/ fos_importer **v1.3**（透過）/ ceo_assistant **v2.0**（§5抽出）。

## 3-2. 起動コマンド設計（追加仕様 2026-07-08）

**CEOは「Morning Brief」という言葉を使う必要はない。朝は「おはよう」だけで十分。**

| CEOの言葉 | AIの解釈 | 実行内容 |
|---|---|---|
| **「おはよう」** | CEO Operating Morning Brief 起動 | ⓪昨夜のAI作業の集計 → ④FOSレビュー（提案生成）→ ①〜⑦のBrief発行 |
| 「Morning Brief」 | 同上（従来コマンドも維持） | 同上 |
| 「仕事始めよう」（将来） | 今日のタスク実行開始 | Briefで承認済みのAI Actions（§2-⑤）を順に実行（Draft/取込/生成のみ） |
| 「今日は終わり」（将来） | 1日のクローズ | Result整理・Learning更新・FOS完了化の依頼リスト提示・翌朝の準備 |

- 自然な会話がAI OSの起動コマンドになる（CEOがシステム用語を覚えない設計）

### 夜間準備思想 — 「朝に生成する」ではなく「朝には完成している」（CEO承認時追加 2026-07-08）

```
夜（AI）: CSV取込 → 同期 → Dashboard更新 → Result更新 → FOSレビュー → Morning Brief下書き
朝（CEO）: 「おはよう」 → 表示だけ
```

- CEOは「おはよう」と言うだけ。AIは**「お待ちしていました」**という状態で迎える
- Brief下書きは前夜に `06_Reports/morning_brief/_draft/` へ生成し、「おはよう」時は**下書きの表示+当朝時点の差分反映（FOS変化・新着のみ）**で数秒で完成させる
- **過渡期ルール**: 夜間自動実行（スケジュール化）が未設定の間は「おはよう」時にその場で生成する。ただし設計思想は常に「朝には完成している」であり、実装はこの状態を目指す
- 夜間処理も全ルール適用（読み取り専用・書込ホワイトリスト・FOS書換禁止・推測禁止）
- 実装: CEO_ASSISTANT.md §2「起動条件」へ「おはよう」追加 + 夜間パイプラインのスケジュール化（ceo_assistant v2.0実装Sprint以降）

## 5. 毎日のパイプライン（将来像・会社を動かす入口）

```
【夜・AI】 Dataset同期（7ソース+FOS）→ CSV取込 → Dashboard更新 → Result更新
          → AIレビュー（異常・変化検出） → FOSレビュー（提案生成）
          → AI実行可能タスク抽出（ai_ready + 代行検出） → Morning Brief下書き
【朝・CEO】「おはよう」 → 表示（+当朝差分） → 5分で判断: 判断する・任せる・放置する・確認する
          → Decision（Decision Log） → Result（Result Layer） → Learning（Insight→…→Knowledge）
```

## 6. CEO Dashboardとの関係（**CEO決定 2026-07-08: 統合しない・役割分離**）

| | **CEO Dashboard** | **Morning Brief** |
|---|---|---|
| 役割 | **会社の現在地を見る画面** | **CEOが朝5分で判断する画面** |
| 開くタイミング | **いつでも**（随時） | **「おはよう」と言った時だけ**生成されるCEO専用レポート |
| 生成 | dashboard_generator（独立維持） | ceo_assistant v2.0 |

- **DashboardをMorning Briefに埋め込むことは可（①Company Status等でDashboardの数値を参照）。ただしDashboard自体を置き換えない**
- dashboard_generatorとceo_assistantは別ツールのまま維持（共通の読取ロジックはモジュール共有可・出力面は分離）

## 7. 不変の制約（v2.0でも変えないもの）

1. 推測禁止（未接続=「未接続」・未入力=null・数値の解釈をAIが確定しない）
2. AIはFOSを書き換えない（レビューは提案のみ）
3. 対外送信・支払・発注・契約の実行はCEOのみ（AI ActionsはDraft/取込/生成まで）
4. draft Knowledgeを判断根拠にしない / Knowledge直行禁止
5. 判断・Result判定・昇格の確定はCEOのみ
6. 書込先: 06_Reports・10_AI_Memory・07_Data（各Importerのホワイトリスト）のみ
