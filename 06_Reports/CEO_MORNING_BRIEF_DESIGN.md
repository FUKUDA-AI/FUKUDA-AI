状態: Released（CEO承認 2026-07-06・Morning Brief第1号・判断1）
関連文書: [../03_Agents/AGENT_COLLABORATION.md](../03_Agents/AGENT_COLLABORATION.md)（日次フロー）/ [../07_Data/DATA_SOURCE_DESIGN.md](../07_Data/DATA_SOURCE_DESIGN.md)

> Morning Briefは毎朝CEOが見る**1枚の経営ブリーフ**。
> 目的はただ一つ: **CEOがその日に判断すべきことを最大3件に絞り、迷わず動ける状態にする。**
> AIは実行しない。判断材料を整理する。実行・送信・発注・価格変更・対外発信はCEO確認後のみ。

---

## 1. 毎朝AIが確認する情報（Brief生成前のチェックリスト）

| 順 | 確認先 | 見るもの |
|---|---|---|
| 1 | 00_MASTER読込順11文書 | 判断の前提（Session Start Protocol） |
| 2 | 10_AI_Memory（CURRENT_STATE / NEXT / PENDING） | 進行中Sprint・保留・レビュー待ち |
| 3 | released Knowledge（knowledge_index.json経由） | 判断根拠。**Draft Knowledgeは通常判断に使わない** |
| 4 | 07_Data実績データ（接続済み分） | 昨日の売上・在庫・資金の変化 |
| 5 | 前日のDecision Log / CEO指示 | 昨日決まったこと・積み残し |
| 6 | カレンダー・催事日程（v2.0で自動化。当面はCEO提供） | 今日の予定と期限 |

## 2. 各Agentが出す日次レポート（Briefの入力）

各Agentは毎朝、**3行以内**の定型ミニレポートをCEO補佐AIへ提出する。「異常なし」も1行で報告する（沈黙と正常を区別するため）。

| Agent | 日次レポート内容 |
|---|---|
| 催事AI | 直近催事の準備状況 / 出店判断待ち案件 / 設営チェック残 |
| so u AI | 未対応の問い合わせ・手紙 / 製作進行 / 特別対応の要否 |
| SUNNY NOMADO AI | 発信予定 / EC反応の特記 / 催事向け準備 |
| 営業AI | 返信待ち商談 / 見積依頼 / 期限が近い案件 |
| 広告・SEO AI | 広告費消化と効率の特記（接続後）/ 発信Draft |
| 発注・在庫AI | 在庫アラート / 発注判断待ち / リードタイム警告 |
| 資金繰りAI | 今週の入出金予定 / 資金残高の余裕度（接続後） |
| 商品企画AI | 進行中企画の状態 / 四問チェック待ち |
| 秘書AI | 今日の予定・期限 / CEOレビュー待ちキューの件数 |
| CEO補佐AI | （統合担当・§3） |

## 3. CEO補佐AIの統合方法

1. **収集**: 全Agentミニレポート + §1のチェック結果を集める
2. **選別**: 「判断が必要なもの」だけを抽出（情報共有と判断事項を分ける）
3. **絞り込み**: §5のルールで最大3件に絞る。落ちた案件は「今日やらないこと」へ明記
4. **整形**: 各判断事項に5点セット（理由・期待効果・リスク・優先順位・実行手順）+ 根拠（EP/KN-ID/数値出典）を添える
5. **検査**: 憲法・CORE/EVOLVINGとの整合を最終確認（値引き提案等が紛れていないか）
6. **発行**: `06_Reports/morning_brief/YYYY-MM-DD.md` として保存し、CEOへ提示

## 4. Morning Briefの表示項目（1枚テンプレート）

```markdown
# CEO Morning Brief — YYYY-MM-DD（曜日）

## 🔴 今日の最重要判断（最大3件）
1. 【カテゴリ】判断してほしいこと1行 → 推奨案 + 理由/効果/リスク/手順（根拠: EP-xxx / KN-xxx）
   CEO判断: [ 承認 / 却下 / 保留 ] コメント: ______
2. …
3. …

## ⛔ 今日やらないこと
- （3件に入らなかった判断・急がない案件。理由つき。勝手に消えていないことを示す）

## ⚠️ 注意（変化があったものだけ表示・なければ「異常なし」1行）
- 資金繰り: …
- 催事: …
- 在庫: …
- 顧客対応: …
- 広告/EC: …

## 📋 レビュー待ち（件数と最優先1件のみ）
- 例: Knowledge Draft 13件（最優先: KN-BRD-0001）

## ⏭ 次にCEOが決めること（今日でなくてよいが近づいている判断）
- …

## 💡 AIからの提案（任意・最大1件）
- 判断要求ではない改善提案・気付き

## 📝 Decision Log候補
- 昨日のCEO判断で未記録のもの → 承認で記録

## ✍️ CEO入力欄
- 各判断への回答（承認/却下/保留 + 一言）
- 今日の優先変更: ______
- AIへの指示・気付き: ______（→ Insight Draft候補として学習サイクルへ）
```

## 5. 判断事項を最大3件に絞るルール

優先スコアの高い順に3件。同点はCEO補佐AIが調停し、落選理由を「今日やらないこと」に書く。

1. **緊急アラート（§6）は無条件で1位**（枠外でも即時通知）
2. **期限**: 今日〜3日以内に締切があるもの（催事契約・入金・納期）
3. **金額×不可逆性**: 動く金額が大きい/後戻りできない判断ほど上位
4. **待ち人数**: お客様・職人・取引先が返事を待っているもの（憲法: 人を待たせない）
5. **判断優先順位**（CORE第2条: 現金化スピード→粗利→ブランド価値→…）で最終同点決着
6. **絞りの哲学**: 迷ったら載せない。「3件全部やれば今日は前進した」と言える構成にする（静けさと余白は経営判断にも適用する）

## 6. 緊急アラート条件（Briefを待たずCEOへ即時通知）

1. 資金: 支払不能リスク・大口入金の遅延（資金繰りAI）
2. 事故・重大クレーム: 催事設営事故・顧客への重大な不手際（現場Agent → 直訴ルート）
3. 供給停止: 職人・工場の稼働停止、主力商品の欠品確定（発注・在庫AI）
4. ブランド毀損: 誤った対外発信・炎上の兆候・知財侵害の発見（各ブランドAI/広告AI）
5. システム: Knowledge/ログの破損・不整合の検知（秘書AI）

アラート時も**AIは対応を実行しない**。事実の時系列整理と一次対応案Draftまで（シナリオ6準拠）。

## 7. CEOが入力する項目

- 各判断への回答: **承認 / 却下 / 保留** + 一言コメント（1行でよい）
- 今日の優先変更（AIの選んだ3件を差し替える指示）
- AIへの指示・気付き（自由記述 → Insight Draft候補として学習サイクルへ還流）
- 所要時間の想定: **5分以内で完了する設計**（読む3分+記入2分）

## 8. Brief後にDecision Logへ残す内容

CEO記入後、秘書AI（当面はFUKUDA AI本体）が以下をDecision Log Draftへ記録する。

- 判断内容（何を承認/却下/保留したか）
- 判断理由（CEOコメント。空欄なら「理由未記入」と正直に記録）
- 根拠にしたEP / Knowledge ID（どの原則が実際に使われたか → **EP運用記録にも追記**）
- 日付・Brief番号
- 却下・保留の場合はその理由（Generator・Brief精度改善の学習データ）

## 9. 日次・週次・月次へのつながり

```
日次 Morning Brief（判断3件・5分）
  ↓ 毎日のCEO判断がDecision Logに蓄積・EP運用記録が育つ
週次レビュー（30分）: 日次では追わない数値トレンド・Sprint進捗・Draft一括レビュー
  ↓ 週次の学び・数値が月次の材料になる
月次総括: 学習サイクル実行（Insight→Pattern→Lesson→Principle）
         EP→CORE昇格検討・ROADMAP見直し
```

日次は「判断」、週次は「レビュー」、月次は「学習と昇格」。同じ情報を二度CEOに見せない（日次で判断済みのものは週次では結果だけ）。

## 10. 段階分け

### v1.1 — 手動運用（今すぐ始められる）
- CEOがセッションを開き「Morning Brief」と言えば、FUKUDA AIが§1〜4を手動実行して発行
- データは10_AI_Memory・レビューキュー・CEO提供情報のみ。Agent未実装分はFUKUDA AI本体が代行
- まず**Briefの型に慣れる**ことが目的

### v1.2 — 半自動（データ接続後）
- 催事売上・Sheets・Shopify接続により、注意セクションが実数値で自動生成
- CEO補佐AI・催事AI・so u AI実装により、ミニレポートが実Agentから集まる
- Decision Log Draftの自動起票

### v2.0 — 完全日次運用
- スケジュール実行で毎朝自動発行（Gmail/Calendar接続で予定・未対応も自動集約）
- 緊急アラートの自動検知・通知
- CEO入力→ 各Agentへの指示展開まで一気通貫（実行は変わらずCEO確認後のみ）
