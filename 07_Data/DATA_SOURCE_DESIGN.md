## 0. 共通ルール（全データソース）

1. **外部データはそのままKnowledgeにしない。** 数値・履歴は07_Dataへ保存し、**意味づけされた学びだけ**が学習サイクル（Insight→…→Lesson）を経てKnowledge候補になる
2. Agentは原則として **Released Knowledge + 必要な実績データ（07_Data）** のみ参照する
3. 個人情報（顧客氏名・住所・連絡先・メール本文等）は**最小限**に扱う。分析には集計値・匿名化データを使う
4. **CEO確認なしに外部への発信・送信・削除・変更をしない。** 接続はすべて読み取り専用から始める
5. 保存先は `07_Data/<ソース名>/`。生データと集計データを分け、取込日時と出典を必ず記録する
6. データ取込機能は新規実装ごとにVersion管理（Development Standard準拠）

## 2. 接続の優先順位

| 順 | データソース | 理由 |
|---|---|---|
| 0 | **FOS**（2026-07-06追加） | **日次運用の中枢**。Morning Brief・タスク・Sprint・PENDINGの入力元であり、毎朝使う。HTMLファイル1つで接続コスト最小（移設のみ要CEO作業） |
| 1 | 催事売上データ | 主要販路。催事AI（実装優先2位）の判断精度に直結。手元データの整理から始められ接続コスト最小。KN-EVT-0001の定量更新にもなる |
| 2 | Google Sheets | 現在の実務管理の実態がここにある。1つの接続で催事・発注・在庫など複数領域のデータが入る「取込口」 |
| 3 | Shopify | EC実績=ブランド別採算の基礎数値。CEO補佐AIの週次サマリーの核 |
| 4 | 発注・在庫データ | 判断優先①現金化スピードの管理に必須（Sheets経由で先行取込→専用化） |
| 5 | 会計・資金繰りデータ | 資金の番人の目。機密性が高いため取扱ルール確定後に接続 |
| 6〜 | Meta広告 → Google Drive → Google Calendar → Instagram → Gmail | 発信系・個人情報系は運用ルールとAgent成熟後（Gmailは最後: 個人情報リスク最大のため） |

## 3. 段階分け（v1.1 / v1.2 / v2.0）

### v1.1 — 数値の土台（読み取りのみ）
- **FOS**（日次運用ボード。移設後すぐ接続・Brief入力の中枢）
- 催事売上データ（過去分の手動整理→07_Data/events/）
- Google Sheets（既存管理表のスナップショット取込）
- Shopify（売上・在庫の日次集計）
- → 催事AI・CEO補佐AIが実績数値つきで判断できる状態

### v1.2 — 資源とカネ（読み取りのみ）
- 発注・在庫データ（専用取込・在庫アラート稼働）
- 会計・資金繰りデータ（週次CF・月次PL）
- Meta広告（週次レポート）
- Google Drive（文書索引）
- → 発注・在庫AI／資金繰りAI／広告・SEO AIが本領発揮

### v2.0 — 日常への組み込み（一部Draft作成まで）
- Gmail（限定ラベルの日次要約・返信Draft）
- Google Calendar（日次フロー起点・予定Draft）
- Instagram（発信効果測定・投稿Draft）
- → 秘書AIの日次フロー完全稼働。全AgentがDraft作成まで担い、実行は常にCEO

## 4. 実装時の必須事項（各接続Sprintで）

1. 接続ごとに「取込機能 vX.X」としてVersion管理・CHANGELOG記録
2. 初回接続時にCEOと「取込範囲・除外項目（個人情報・機密）」を確認してから実装
3. 07_Data配下のREADMEにデータ辞書（項目・更新頻度・出典）を作成
4. knowledge_index.jsonと同様、実績データにも索引を設ける（Agentの索引ファースト参照）
